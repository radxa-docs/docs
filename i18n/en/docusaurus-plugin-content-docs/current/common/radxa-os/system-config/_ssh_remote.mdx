## Troubleshooting

If you encounter issues during SSH connection, try the following troubleshooting steps.

### Check SSH Service Status

If you are connecting remotely (over LAN, not via direct display), ensure the SSH service is running:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo systemctl status ssh
```

</NewCodeBlock>

If the service is not running, start it with:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo systemctl start ssh
sudo systemctl enable ssh
```

</NewCodeBlock>

:::tip
If the `ssh` service name does not work, try `sshd`: `sudo systemctl status sshd`
:::

### Check SSH Configuration

Ensure SSH configuration allows password authentication:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo grep -v "^#" /etc/ssh/sshd_config | grep PasswordAuthentication
```

</NewCodeBlock>

Expected output should be `PasswordAuthentication yes`. If not, edit the configuration file:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo nano /etc/ssh/sshd_config
```

</NewCodeBlock>

Find and ensure the following line is uncommented and set to `yes`:

```
PasswordAuthentication yes
```

Save and restart the SSH service:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo systemctl restart ssh
```

</NewCodeBlock>

### Check Network and Firewall

Ensure the firewall is not blocking SSH connections (default port 22):

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
# Check iptables rules
sudo iptables -L -n | grep -E "22|ssh"

# or check nftables
sudo nft list ruleset 2>/dev/null | grep -E "22|ssh"
```

</NewCodeBlock>

### Verify IP Address

Ensure you are using the correct IP address:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
ip addr show | grep inet
```

</NewCodeBlock>

You can also confirm the device IP from your PC:

<NewCodeBlock tip="PC$" type="host">

```bash
# Windows
arp -a | findstr "192.168"

# Linux/macOS
arp -a | grep 192.168
```

</NewCodeBlock>

### Check SSH Logs

View SSH service logs for detailed error information:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo journalctl -u ssh -b --no-pager | tail -30
```

</NewCodeBlock>

### Common Issues

#### Incorrect Password

Ensure you are using the correct username and password. Radxa OS default username is `radxa`, password is `radxa`.

If you forgot the password, you can reset it via serial connection or by connecting a display:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
sudo passwd radxa
```

</NewCodeBlock>

#### SSH Public Key Permission Issues

If you previously configured SSH public key authentication, permission errors may cause login failures. Check server-side permissions:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
# Check .ssh directory permissions
ls -la ~/.ssh

# Correct permissions should be 700
chmod 700 ~/.ssh

# Check authorized_keys permissions (should be 600)
chmod 600 ~/.ssh/authorized_keys
```

</NewCodeBlock>

#### Windows SSH Client Problems

If you encounter issues with Windows SSH client:

1. Try Windows built-in SSH client (PowerShell or CMD)
2. Try [Tabby](https://tabby.sh/), [PuTTY](https://www.putty.org/), or [MobaXterm](https://mobaxterm.mobatek.net/)
3. Check Windows Firewall settings to ensure SSH connections are not blocked

#### Connection Timeout

1. Confirm device and PC are on the same LAN
2. Confirm the IP address is correct (device IP cannot be `127.0.0.1`)
3. Try ping the device IP to verify network connectivity

<NewCodeBlock tip="PC$" type="host">

```bash
ping 192.168.x.x
```

</NewCodeBlock>

### Reset SSH Configuration

:::warning
The following operations will interrupt your SSH connection. If you are connecting remotely, ensure you have an alternative access method (such as serial connection) before proceeding.
:::

If none of the above steps resolve the issue, you can reset the SSH configuration:

<NewCodeBlock tip={`radxa@${props?.board ?? 'device'}$`} type="device">

```bash
# Backup current configuration
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.backup

# Restore default configuration
sudo dpkg-reconfigure openssh-server
```

</NewCodeBlock>

Restart and reconfigure your SSH connection after completion.

### Get Help

If the issue persists, please submit a GitHub Issue with the following information:

- SSH service status output (`sudo systemctl status ssh`)
- `PasswordAuthentication` setting in SSH configuration
- Firewall status
- Complete error message when attempting to connect from the client
- SSH logs (`sudo journalctl -u ssh -b --no-pager | tail -30`)
